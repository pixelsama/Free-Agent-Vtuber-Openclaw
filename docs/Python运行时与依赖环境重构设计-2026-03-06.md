# Python 运行时与依赖环境重构设计（2026-03-06）

## 1. 背景

当前桌面端的 Python 能力存在一个结构性问题：

1. Python 解释器被当成 ASR/TTS bundle 的附属物。
2. `Nanobot` 运行时会记录并复用某个语音 bundle 中的 Python 路径。
3. 当用户删除语音模型或清空 `voice-models` 后，`Nanobot` 可能因解释器路径失效而无法启动。
4. Python 依赖安装与模型文件下载混在同一层，缺少清晰的生命周期边界。

这次重构的目标，是把 Python 从“模型附属品”提升为“应用级基础设施”，同时把依赖隔离从“耦合在单个 bundle 内”改为“按能力栈隔离”。

## 2. 目标

本次设计希望达成以下结果：

1. `Python runtime` 属于整个桌面应用，不依附任何单个 ASR/TTS/Nanobot 资源。
2. 任何需要 Python 的功能在安装时，都先检查共享 runtime 是否存在。
3. 不同能力栈使用相互隔离的 Python env，避免依赖冲突。
4. 模型文件、源码仓库、Python env、Python runtime 四层解耦，删除时互不误伤。
5. 默认 Python 版本从 `3.11` 升级到 `3.12`，保留为个别 profile 回退到 `3.11` 的能力。

## 3. 非目标

本阶段不做以下事情：

1. 不追求一次性重写所有 ASR/TTS/Nanobot 逻辑。
2. 不默认把每一个模型都拆成独立 env。
3. 不引入“多 Python 主版本长期并存”的复杂 UI 管理。
4. 不在本阶段改变现有 renderer 层的 IPC 语义。

## 4. 当前问题梳理

现状可以概括为三类耦合：

### 4.1 解释器耦合

语音 bundle 自带 Python，`Nanobot` 又复用了语音 runtime 中的解释器路径。  
这导致：

1. 删除语音 bundle 后，聊天后端也可能损坏。
2. Python 生命周期跟模型生命周期错误绑定。

### 4.2 依赖耦合

`mlx-audio`、`funasr`、`edge-tts`、`nanobot` 等 Python 依赖当前没有清晰隔离边界。  
未来只要某个库升级或引入不兼容版本，就可能影响其他能力。

### 4.3 安装器职责耦合

当前“下载模型”往往隐含了：

1. 下载 Python runtime
2. 安装 pip 依赖
3. 下载模型文件
4. 写入运行时状态

这些步骤缺乏明确分层，后续很难做复用、修复和清理。

## 5. 设计原则

重构遵循以下原则：

1. 共享解释器，隔离依赖。
2. runtime、env、artifact 分层管理。
3. 删除下层资源时，不应影响上层共享资源。
4. 安装流程统一走 `runtime -> env -> artifact`。
5. 依赖隔离按“能力栈 / 依赖锁”优先，而不是按“模型名”优先。

## 6. 推荐总体架构

### 6.1 四层资源模型

建议把 Python 相关资源拆为四层：

1. `Python runtime`
2. `Python env`
3. `Artifact`
4. `Bundle / Runtime state`

语义如下：

1. `Python runtime`
   - 一份共享 CPython。
   - 只提供解释器，不携带业务依赖。
2. `Python env`
   - 基于共享 runtime 创建的隔离虚拟环境。
   - 负责安装某一能力栈所需依赖。
3. `Artifact`
   - 模型文件、tokenizer、Nanobot repo、脚本等资源。
4. `Bundle / Runtime state`
   - 负责记录“哪个功能正在引用哪个 env / artifact”。

### 6.2 Manager 分层

建议新增或重构为以下 manager：

1. `PythonRuntimeManager`
2. `PythonEnvManager`
3. `ArtifactManager`
4. `DependencyResolver`

职责如下：

#### PythonRuntimeManager

负责：

1. 下载共享 CPython。
2. 校验解释器可执行性。
3. 返回稳定的 `pythonBin`。
4. 升级和修复 runtime。

不负责：

1. 业务依赖安装。
2. 模型文件下载。

#### PythonEnvManager

负责：

1. 基于共享 runtime 创建 env。
2. 按 profile 或 lock 安装 pip 依赖。
3. 校验 env 的健康状态。
4. 输出 `envPythonExecutable`。

#### ArtifactManager

负责：

1. 下载 ASR/TTS 模型 bundle。
2. 下载 tokenizer 等附属资源。
3. 下载和维护 `Nanobot` repo。

#### DependencyResolver

负责：

1. 读取 manifest。
2. 判断某个安装项是否需要 Python。
3. 解析它需要哪个 env profile。
4. 组织安装顺序并返回最终运行时配置。

## 7. 目录结构设计

建议统一放在应用用户数据目录下，例如：

```text
~/Library/Application Support/free-agent-vtuber-openclaw-desktop/
  python-runtime/
    python/
    state.json
  python-envs/
    <env-id>/
      bin/
      lib/
      manifest.json
  voice-models/
    bundles/
      <bundle-id>/
    state.json
  nanobot-runtime/
    repo/
    state.json
```

说明：

1. `python-runtime/python/` 只放共享 Python。
2. `python-envs/<env-id>/` 只放 env 和该 env 的 manifest。
3. `voice-models/bundles/<bundle-id>/` 只放模型和相关资源，不再放 Python。
4. `nanobot-runtime/repo/` 只放 Nanobot repo，不再把 Python 路径作为唯一真值持久化。

## 8. env 隔离粒度

### 8.1 不建议默认“每模型一个 env”

原因：

1. 磁盘占用会显著增大。
2. 安装时间会明显增加。
3. 大量模型只是权重不同，依赖并不不同。
4. 后续维护和清理复杂度会失控。

### 8.2 推荐“按能力栈 / 依赖锁隔离”

建议初始 profile：

1. `nanobot-default`
2. `tts-qwen3-mlx`
3. `asr-funasr`
4. `tts-edge`

后续若出现依赖冲突，再进一步细分：

1. `tts-qwen3-mlx-v2`
2. `asr-funasr-py311`

### 8.3 env-id 生成策略

推荐：

```text
env-id = hash(profile + pythonVersion + lockHash + platform)
```

好处：

1. 依赖相同的模型和能力栈可复用 env。
2. 依赖变化时会自动分裂出新 env，而不是污染旧 env。
3. 可以自然支持按 Python 小版本并存。

## 9. Manifest 设计

每个可安装单元都应声明自己的依赖关系。

### 9.1 语音模型 manifest 示例

```json
{
  "id": "builtin-tts-qwen3-0.6b-8bit-v1",
  "type": "voice-model",
  "provider": "python",
  "capability": "tts",
  "requiresPython": true,
  "pythonEnvProfile": "tts-qwen3-mlx",
  "pythonVersion": "3.12",
  "artifacts": {
    "modelId": "mlx-community/Qwen3-TTS-12Hz-0.6B-CustomVoice-8bit"
  }
}
```

### 9.2 Nanobot manifest 示例

```json
{
  "id": "builtin-nanobot-runtime-v1",
  "type": "backend-runtime",
  "provider": "python",
  "capability": "nanobot",
  "requiresPython": true,
  "pythonEnvProfile": "nanobot-default",
  "pythonVersion": "3.12",
  "artifacts": {
    "repoArchiveUrl": "https://codeload.github.com/HKUDS/nanobot/tar.gz/refs/heads/main"
  }
}
```

### 9.3 建议字段

manifest 至少应支持：

1. `id`
2. `type`
3. `provider`
4. `capability`
5. `requiresPython`
6. `pythonEnvProfile`
7. `pythonVersion`
8. `artifacts`

```json
  "pythonVersion": "3.12.9",
  "installedAt": "2026-03-06T00:00:00.000Z"
}
```

### 10.2 python-envs/<env-id>/manifest.json

```json
{
  "version": 1,
  "envId": "tts-qwen3-mlx-py312-abc123",
  "profile": "tts-qwen3-mlx",
  "pythonVersion": "3.12",
  "lockHash": "abc123",
  "pythonExecutable": ".../python-runtime/python/bin/python3",
  "envPythonExecutable": ".../python-envs/tts-qwen3-mlx-py312-abc123/bin/python",
  "installedAt": "2026-03-06T00:00:00.000Z",
  "lastUsedAt": "2026-03-06T00:00:00.000Z"
}
```

### 10.3 voice-models/state.json

```json
{
  "version": 1,
  "installedBundles": [
    {
      "bundleId": "builtin-tts-qwen3-0.6b-8bit-v1",
      "path": ".../voice-models/bundles/builtin-tts-qwen3-0.6b-8bit-v1",
      "pythonEnvId": "tts-qwen3-mlx-py312-abc123"
    }
  ]
}
```

### 10.4 nanobot-runtime/state.json

```json
{
  "version": 1,
  "repoPath": ".../nanobot-runtime/repo",
  "pythonEnvId": "nanobot-default-py312-def456",
  "installedAt": "2026-03-06T00:00:00.000Z"
}
```

关键要求：

1. `Nanobot` 不再把 `pythonExecutable` 当成最终真值持久化。
2. `pythonExecutable` 的解析应由 `PythonRuntimeManager / PythonEnvManager` 提供。
3. 功能状态里只记录 `pythonEnvId`。

## 11. 安装流程设计

统一安装顺序：

1. 读取 manifest。
2. 如果 `requiresPython = true`，执行 `ensureRuntime(pythonVersion)`。
3. 执行 `ensureEnv(pythonEnvProfile, pythonVersion)`。
4. 下载并安装 artifact。
5. 写入引用关系和状态。

### 11.1 安装 TTS 模型

以 Qwen3 TTS 为例：

1. 确认共享 `Python 3.12` 是否存在。
2. 若不存在，先下载并安装 runtime。
3. 确认 `tts-qwen3-mlx` env 是否存在。
4. 若不存在，创建 env 并安装 `mlx-audio` 等依赖。
5. 下载 TTS 模型文件。
6. 将 bundle 与 `pythonEnvId` 绑定。

### 11.2 安装 Nanobot

1. 确认共享 `Python 3.12` 是否存在。
2. 若不存在，先安装 runtime。
3. 确认 `nanobot-default` env 是否存在。
4. 创建 env 并安装 `nanobot` 需要的依赖。
5. 下载或更新 `nanobot` repo。
6. 在 `nanobot-runtime/state.json` 中记录 `pythonEnvId`。

## 12. 删除与清理流程

### 12.1 删除模型

删除某个 ASR/TTS bundle 时：

1. 删除 bundle 自身文件。
2. 删除 bundle 对 `pythonEnvId` 的引用。
3. 不删除共享 Python runtime。
4. 不直接删除 env。

### 12.2 删除 Nanobot

1. 删除 repo。
2. 删除对 `nanobot-default` env 的引用。
3. 不删除共享 Python runtime。

### 12.3 清理孤儿 env

当某个 env 没有任何引用时：

1. 将其标记为“可清理”。
2. 可立即删除，或保留一段宽限期后再删除。
3. 建议根据 `lastUsedAt` 实现延迟清理，避免频繁重建。

## 13. Python 版本策略

### 13.1 推荐默认版本：Python 3.12

推荐把项目默认共享 runtime 从 `3.11` 升到 `3.12`，原因如下：

1. `3.12` 比 `3.11` 更新，生命周期更长。
2. `3.13` 相比 `3.12` 更新得更激进，第三方兼容性风险更高。
3. 当前依赖栈里，`Nanobot` 已明确支持 `>=3.11`，并在包元数据中标注了 `3.11` 与 `3.12`。
4. `mlx-audio` 与 `edge-tts` 对 `3.12` 的兼容风险较低。

### 13.2 为什么不把 3.13 设为默认

原因：

1. 对桌面应用来说，稳定性优先于“最新”。
2. 某些依赖对 `3.13` 的测试覆盖尚不充分。
3. 本项目目前同时涉及 `mlx-audio`、`funasr`、`nanobot` 等多套生态，保守一些更稳。

### 13.3 为什么不继续固定 3.11

原因：

1. `3.11` 依然可用，但已不再是更优默认值。
2. 若现在继续把共享 runtime 定在 `3.11`，未来仍需再做一次主版本迁移。
3. 既然要做架构重构，适合在同一阶段把默认值切到 `3.12`。

### 13.4 兼容性例外

默认策略应是：

1. 全局默认 runtime 为 `3.12`
2. env profile 允许单独声明 `pythonVersion`
3. 若某个 profile 经验证只能稳定运行在 `3.11`，允许单独回退

例如：

1. `tts-qwen3-mlx -> 3.12`
2. `nanobot-default -> 3.12`
3. `asr-funasr -> 如有必要可临时保留 3.11`

## 14. 现有模块改造建议

### 14.1 需要新增或重构的模块

建议新增：

1. `desktop/electron/services/python/pythonRuntimeManager.js`
2. `desktop/electron/services/python/pythonEnvManager.js`
3. `desktop/electron/services/python/dependencyResolver.js`

建议重构：

1. `desktop/electron/services/voice/voiceModelCatalog.js`
2. `desktop/electron/services/voice/voiceModelLibrary.js`
3. `desktop/electron/services/chat/nanobot/nanobotRuntimeManager.js`

### 14.2 voiceModelCatalog 改造方向

从现在的“runtime + model”混合定义，改为：

1. 描述模型自身 artifact
2. 描述所需 env profile
3. 不再把 Python runtime 打包为 bundle 内容

### 14.3 voiceModelLibrary 改造方向

从“安装 bundle 时顺便装 Python”改为：

1. 调用 `DependencyResolver`
2. 先 `ensureRuntime`
3. 再 `ensureEnv`
4. 再装模型
5. 最终只回填 bundle 与 env 的关系

### 14.4 NanobotRuntimeManager 改造方向

从“自己记住一个 Python 路径”改为：

1. 只负责 repo 下载和状态
2. 通过 `pythonEnvId` 解析运行 Python
3. 启动时从 `PythonEnvManager` 获取最终 `envPythonExecutable`

## 15. 迁移策略

建议分四步迁移，避免一次性大改：

### 阶段 1：独立共享 Python runtime

目标：

1. 把解释器从语音 bundle 中拆出来
2. 修复 `Nanobot` 与语音模型路径耦合问题

### 阶段 2：引入 PythonEnvManager

目标：

1. 先支持两个 profile
   - `nanobot-default`
   - `tts-qwen3-mlx`
2. 跑通 env 创建、依赖安装和路径解析

### 阶段 3：模型安装器改造

目标：

1. 让 ASR/TTS/Nanobot manifest 声明 env 依赖
2. 安装流程统一改为 `runtime -> env -> artifact`

### 阶段 4：旧状态迁移与修复

目标：

1. 识别历史 `pythonExecutable` 字段
2. 迁移为 `pythonEnvId`
3. 对失效旧路径执行自动修复

## 16. 旧数据兼容策略

启动时建议执行以下兼容逻辑：

1. 检查 `Nanobot` 状态中是否仍保存旧的 `pythonExecutable`
2. 若路径命中 `voice-models/bundles/.../runtime/python/...`
   - 判定为旧架构残留
3. 执行自动修复：
   - `ensureRuntime(3.12)`
   - `ensureEnv('nanobot-default', 3.12)`
   - 回写 `pythonEnvId`
   - 清理旧 `pythonExecutable`

这样可以避免用户删除语音模型后再次触发 `Nanobot 启动失败`。

## 17. 风险与权衡

### 17.1 复杂度上升

引入共享 runtime 和多 env 后，管理复杂度一定会上升。  
但这是换取稳定生命周期和依赖隔离的必要成本。

### 17.2 磁盘占用

多 env 会比单 bundle 模式占用更多空间。  
但通过“按能力栈 / lock 复用 env”，可以显著低于“每模型一个 env”的方案。

### 17.3 迁移期双轨状态

在迁移过程中，一段时间内可能同时存在：

1. 旧式 bundle 内嵌 Python
2. 新式共享 runtime + env

因此必须补齐迁移逻辑和兼容判断。

## 18. 结论

本次重构建议采用以下最终方案：

1. 一份应用级共享 Python runtime
2. 多份隔离 env，但按能力栈和依赖锁复用
3. 模型文件与 repo 只管理 artifact，不再携带解释器
4. 安装器统一走 `runtime -> env -> artifact`
5. 默认 Python 版本升级为 `3.12`
6. 仅当个别能力栈明确不兼容时，才允许 profile 级回退到 `3.11`

这套方案能同时解决：

1. `Nanobot` 与 ASR/TTS 的解释器路径耦合问题
2. Python 依赖冲突问题
3. 安装、删除、修复流程混乱的问题
4. 后续继续扩展 Python 能力时的架构可维护性问题

## 19. 后续建议

建议后续实现阶段优先完成：

1. `PythonRuntimeManager`
2. `Nanobot` 运行时从 `pythonExecutable` 切换到 `pythonEnvId`
3. `voiceModelCatalog` manifest 化
4. `PythonEnvManager`
5. 旧状态自动修复
