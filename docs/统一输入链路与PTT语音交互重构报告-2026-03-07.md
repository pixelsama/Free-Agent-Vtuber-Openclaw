# 统一输入链路与 PTT 语音交互重构报告（2026-03-07）

## 1. 报告目的

本报告整合以下讨论结论，形成一版可执行的架构方案：

1. 语音输入与文字输入在 Nanobot 前的链路收敛现状
2. 现有 `session`/`voice session` 设计导致的冲突与问题
3. 新语音交互方式（长按说话、松手提交一次请求）的可行方案

本报告只做设计与路径建议，不包含代码变更。

---

## 2. 现状结论（基线）

### 2.1 输入链路收敛：后端层已收敛

当前项目中，文字输入和语音输入在进入 Nanobot 前都会收敛到 `chat:stream`，最终以文本 `content` 发送到 Nanobot。

- 文字输入：前端 `startStreaming -> chat:stream:start`
- 语音输入：`ASR final -> onAsrFinal -> chat:stream:start`

结论：对 Nanobot 推理核心来说，本质都是文本请求。

### 2.2 现有主要问题：前端事件入口和语音会话依赖未收敛

已确认的两个现象：

1. 文字输入常常不触发 TTS  
   原因：TTS 依赖 `voice session`，文本输入本身不保证存在活动 voice session。
2. 语音输入触发 TTS 但字幕不显示  
   原因：`chat:stream:event` 监听器存在懒初始化路径，纯语音自动流时可能未建立监听，字幕桥拿不到分段事件。

结论：后端发送链路收敛了，但前端消费链路和生命周期管理仍分裂。

---

## 3. Session 设计问题（架构冲突）

### 3.1 冲突本质

当前同时存在：

1. chat stream 的流生命周期
2. voice session 的音频/ASR/TTS 生命周期

两者耦合点较多，但启动时机不同、事件入口不同，导致：

1. 同一个用户意图在不同路径下行为不一致（是否播 TTS、是否出字幕）
2. 并发策略不统一（文本偏“抢占中断”，语音路径可能与其不同步）
3. 状态可见性复杂（UI 对 `stream`、`voice`、`tts` 需要多处拼接）

### 3.2 产品侧不匹配

你的场景不是严格问答回合同步场景，而是可能存在：

1. 长链任务
2. 后台主动消息
3. 非用户触发输出

因此“每次都强调短生命周期 session”的收益较低，复杂度和不稳定性更高。

---

## 4. 新语音交互方案评估（PTT：按住说话，松手提交）

## 4.1 方案描述

目标交互：

1. 用户按住快捷键开始说话（PTT）
2. 松开快捷键视为“本次语音输入结束”
3. 按住期间 VAD 继续工作，只负责切段和过滤无声
4. 按住期间 ASR 可提前处理每个切段
5. 松开后等待剩余 ASR 完成，将文本按顺序拼接，仅发一次 Nanobot 请求

### 4.2 该方案的优点

1. 用户意图边界清晰：由按键松开决定“何时提交”
2. 避免自然停顿误触发多请求
3. 可降低尾延迟：ASR 在按住期间已经并行进行
4. 对现有前后端改造规模可控

### 4.3 关键注意事项

1. `Fn` 键不应作为唯一方案  
   在 Electron/浏览器层，`Fn` 常不能稳定作为标准按键事件获取。建议“可配置 PTT 键”，默认用可可靠捕获的键位组合。
2. 需要定义“松手后 flush 超时”  
   防止单段 ASR 卡住导致整次提交卡死。
3. 需要保序拼接  
   ASR 分段并发时必须按段序号拼接文本。

## 5. 目标架构（建议）

### 5.1 总体原则

1. 输入统一：文本和语音最终都走同一个 `submitUserText(...)` 入口
2. 输出统一：无论用户触发还是后台主动消息，统一走 `conversation:event`
3. session 内部化：不向 UI 暴露复杂 `voice session` 语义，仅保留必要 `turnId`/`requestId`

### 5.2 组件建议

1. `ConversationRuntime`（主进程常驻单实例）
   - 负责请求调度、取消策略、事件总线
2. `VoiceInputWindow`（PTT 窗口聚合器）
   - 负责按住期间切段、ASR 任务管理、松手合并提交
3. `SubtitleConsumer` 与 `TtsConsumer`
   - 订阅同一条 `conversation:event`
   - 去除“字幕依赖某路径先初始化”的前提

### 5.3 并发策略建议

对聊天请求采用明确单策略（建议默认）：

1. `latest-wins`（最新优先）  
   新请求到来时中断旧请求；UI 行为更可预期。
2. 可选扩展：`queue`（串行队列）  
   用于特殊工作流，不作为默认。

---

## 6. PTT 详细时序（建议实现）

1. `PTT_KEY_DOWN`
   - 打开“语音输入窗口”
   - 启动录音/VAD
2. `onSpeechEnd(segmentAudio)`
   - 写入 `segmentId`
   - 立即发起 `ASR(segmentId)`（可设置并发上限）
3. `PTT_KEY_UP`
   - 停止收音
   - 触发 `flush`：补齐尾段、等待在途 ASR（带超时）
4. `flush完成`
   - 按 `segmentId` 拼接文本
   - 空文本则丢弃
   - 非空文本仅调用一次 `submitUserText(text, source='voice-ptt')`

## 7. 分阶段改造路径

### Phase 0（快速止血）

1. 将 `chat:stream:event` 监听改为应用启动即常驻
2. 保证语音自动流也可稳定驱动字幕链路

预期：先消除“语音有 TTS 无字幕”的明显异常。

### Phase 1（交互升级）

1. 引入 PTT 输入窗口（按住说话、松手提交）
2. VAD 仅分段，ASR 分段处理但不分段提交
3. 增加保序拼接与 flush 超时

预期：显著减少自然停顿导致的多请求误触发。

### Phase 2（架构收口）

1. 收敛输入入口为统一 `submitUserText`
2. `voice session` 语义内聚到 `ConversationRuntime`，不再作为 UI 主流程依赖
3. 统一输出事件总线（含后台主动消息）

预期：降低状态复杂度，提高后续功能扩展稳定性。

---

## 8. 验收标准（建议）

1. 文字输入：
   - 稳定显示字幕
   - TTS 是否播放由配置策略决定且行为一致
2. 语音 PTT 输入：
   - 一次按住/松手最多触发一次聊天请求
   - 中间自然停顿不会产生额外请求
   - 松手后在设定超时内必有结果（成功或可解释失败）
3. 混合并发：
   - 文本/语音请求冲突时策略明确且可观察（latest-wins 或 queue）
4. 背景主动消息：
   - 不依赖用户输入即可进入统一输出链路（字幕/TTS 同步策略一致）

---

## 9. 风险与控制

1. 风险：PTT 键位跨平台行为差异  
   控制：提供可配置热键和降级方案，不绑定 `Fn` 单键。
2. 风险：ASR flush 等待导致“松手后迟滞”  
   控制：设置并发上限与总超时，超时时使用已完成片段提交并给出提示。
3. 风险：重构期双链路并存引入回归  
   控制：按 Phase 拆分，小步上线，每步补充回归测试。

---

## 10. 结论

建议采用“**PTT 决定提交边界、VAD 仅做切段、ASR 分段提前处理、松手一次性提交**”的方案，并同步推进“输入/输出链路统一化”与“session 内部化”。

这个方向既满足你的产品直觉（减少复杂会话管理），也能直接解决当前最突出的两类问题：

1. 语音自然停顿导致多请求
2. TTS 与字幕链路行为不一致
