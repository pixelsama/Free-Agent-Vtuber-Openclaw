# Nanobot 后端接入技术指导（2026-03-05）

## 1. 背景与目标

当前项目聊天链路强绑定 OpenClaw：

1. 配置项仅有 `OpenClaw Base URL / Token / Agent ID`。
2. 主进程流式调用固定走 `/v1/chat/completions`。
3. 错误码、设置页文案、测试用例都以 `openclaw_*` 为中心。

这导致用户必须先部署 OpenClaw 才能使用聊天功能。对非开发者用户，这一门槛很高。

本次目标：

1. 在项目内增加可选后端框架 `nanobot`。
2. 用户可在设置页切换 `OpenClaw` 或 `Nanobot`。
3. 启用 `Nanobot` 时，不再依赖 OpenClaw 服务。
4. 保持前端现有事件语义不变：`text-delta / done / error`。

## 2. 非目标（本阶段）

1. 不替换现有 OpenClaw 链路，必须保留兼容。
2. 不在本阶段重构 Live2D、语音链路业务逻辑。
3. 不追求一次性做到多后端插件市场，先实现 `openclaw + nanobot` 双后端。

## 3. 现状耦合点梳理

以下文件与 OpenClaw 强耦合，后续改造需要逐步解耦：

1. 主进程请求适配：`desktop/electron/services/openclawClient.js`
2. IPC 流式路由：`desktop/electron/ipc/chatStream.js`
3. 设置测试接口：`desktop/electron/ipc/settings.js`
4. 设置存储：`desktop/electron/services/settingsStore.js`
5. 前端设置 Hook：`front_end/src/hooks/settings/useOpenClawSettings.js`
6. 设置面板：`front_end/src/components/config/ConfigDrawer.jsx`
7. 错误归一化：`front_end/src/utils/normalizeErrorMessage.js`
8. 国际化文案：`front_end/src/i18n/I18nContext.jsx`
9. 前后端测试：`desktop/electron/tests/*` 与 `front_end/tests/*`

## 4. 推荐总体方案

### 4.1 架构原则

1. 保留 Electron Main 作为唯一“后端适配层”。
2. 抽象统一接口 `ChatBackendAdapter`，把 OpenClaw 变成一个实现。
3. 新增 `NanobotBackendAdapter`，在主进程内通过 Python Sidecar 调用 nanobot。
4. UI 和流式 Hook 不感知具体后端，只感知统一事件。

### 4.2 目标架构

```text
Renderer (React)
  <-> preload IPC
  <-> Electron Main
      <-> OpenClawBackendAdapter -> OpenClaw HTTP SSE
      <-> NanobotBackendAdapter  -> Python Sidecar -> nanobot AgentLoop
```

### 4.3 为什么不直接把 nanobot 当成 OpenAI API 来调

`nanobot` 本身是 Agent 框架，不是现成与 OpenClaw 完全对齐的 HTTP 服务。  
其稳定入口是 Python 侧 `AgentLoop.process_direct(...)`。因此推荐走 Sidecar 适配，避免在 Electron 主进程里硬塞完整 Python 逻辑。

## 5. 目录与模块设计

建议新增目录：

```text
desktop/electron/services/chat/
  backends/
    base.js
    openclawBackend.js
    nanobotBackend.js
  backendManager.js
  nanobot/
    nanobotBridgeClient.js
    nanobot_bridge.py
```

说明：

1. `openclawBackend.js` 封装现有 `openclawClient.js`，做到老逻辑复用。
2. `nanobotBackend.js` 负责把 nanobot 结果映射为统一事件。
3. `backendManager.js` 根据 settings 选择适配器并管理生命周期。
4. `nanobot_bridge.py` 处理 Python 侧 AgentLoop 调用、事件输出、取消。

## 6. 统一后端接口定义

在 `base.js` 定义：

```js
/**
 * ChatBackendAdapter contract
 * - validateSettings(settings) -> throws typed error
 * - testConnection(settings) -> { ok, latencyMs? } | throws
 * - startStream({ settings, sessionId, content, options, signal, onEvent }) -> Promise<void>
 * - mapError(error) -> { code, message, status? }
 * - dispose() -> Promise<void>
 */
```

统一事件结构：

```js
{ type: 'text-delta', payload: { content, source } }
{ type: 'done', payload: { source, aborted? } }
{ type: 'error', payload: { code, message, status?, source } }
```

`source` 允许值：

1. `openclaw`
2. `nanobot`

## 7. 设置模型改造（关键）

### 7.1 新 settings 结构

建议把设置升级为：

```json
{
  "chatBackend": "openclaw",
  "openclaw": {
    "baseUrl": "http://127.0.0.1:18789",
    "agentId": "main"
  },
  "nanobot": {
    "enabled": false,
    "workspace": "",
    "provider": "openrouter",
    "model": "anthropic/claude-opus-4-5",
    "apiBase": "",
    "maxTokens": 4096,
    "temperature": 0.2,
    "reasoningEffort": ""
  }
}
```

密钥建议继续放 keytar：

1. `openclaw.token`
2. `nanobot.apiKey`

### 7.2 迁移策略

在 `SettingsStore.init()` 中执行自动迁移：

1. 检测旧版 `openclaw-settings.json`。
2. 迁移到新结构并保留原 token。
3. 默认 `chatBackend=openclaw`，保证升级无感。

## 8. IPC 协议改造

## 8.1 保持兼容

保留现有通道名：

1. `chat:stream:start`
2. `chat:stream:abort`
3. `settings:get`
4. `settings:save`
5. `settings:test`

避免前端大面积改动。

## 8.2 请求参数扩展

`chat:stream:start` 支持可选字段：

```json
{
  "sessionId": "s1",
  "content": "你好",
  "backend": "nanobot",
  "options": {}
}
```

解析规则：

1. 请求带 `backend` 时优先使用请求值。
2. 否则使用 `settings.chatBackend`。

## 8.3 settings:test 行为

`settings:test` 根据 `chatBackend` 分流：

1. `openclaw` -> 复用原 `testOpenClawConnection`。
2. `nanobot` -> 调用 `testNanobotConnection`（检查 runtime + provider + 最小调用）。

## 9. Nanobot 适配器设计

### 9.1 Sidecar 通信模型

建议使用 `stdin/stdout` JSON Lines 协议：

主进程发送：

```json
{"type":"start","requestId":"r1","sessionId":"s1","content":"hello","config":{...}}
{"type":"abort","requestId":"r1"}
```

Sidecar 回传：

```json
{"type":"event","requestId":"r1","event":{"type":"text-delta","payload":{"content":"..."}}}
{"type":"event","requestId":"r1","event":{"type":"done","payload":{"source":"nanobot"}}}
{"type":"event","requestId":"r1","event":{"type":"error","payload":{"code":"nanobot_runtime_error","message":"..."}}}
```

### 9.2 Python bridge 核心职责

1. 加载或复用 `nanobot` provider 与 `AgentLoop`。
2. 调用 `process_direct(content, session_key, ...)`。
3. 将 `on_progress` 回调转为 `text-delta`。
4. 完成时发 `done`。
5. 处理取消和异常映射。

### 9.3 事件映射建议

由于 `nanobot` 默认不是 token 级 SSE，建议先做稳定版映射：

1. `on_progress` 文本 -> `text-delta`（粗粒度）。
2. 最终 `response` 若未完整输出 -> 追加一次 `text-delta`。
3. 最后统一 `done`。

这能保证前端显示流式体验，同时减少首次改造复杂度。

### 9.4 错误码规范（新增）

建议新增并统一前端翻译：

1. `nanobot_missing_config`
2. `nanobot_runtime_not_ready`
3. `nanobot_boot_failed`
4. `nanobot_provider_unavailable`
5. `nanobot_model_call_failed`
6. `nanobot_unreachable`

## 10. 安全与权限控制（必须）

`nanobot` 默认会注册文件、命令执行、网络等工具。接入桌面端前必须做安全收敛：

1. 默认开启 `restrict_to_workspace=true`。
2. 使用专用 workspace：`<appData>/nanobot-workspace`。
3. 首版建议禁用高风险工具（尤其 shell exec）。
4. 最大迭代数、最大 token、超时必须配置上限。
5. 前端给出“本地工具权限”明确提示。

建议在 `nanobot` 分支增加工具开关配置，再集成到本项目。

## 11. UI 与前端改造

### 11.1 设置页

将“OpenClaw 设置”升级为“聊天后端设置”：

1. 后端选择器：`OpenClaw / Nanobot`。
2. 不同后端展示不同表单字段。
3. 保留“保存设置 / 连接测试 / 清除密钥”。

### 11.2 Hook 重命名

建议：

1. `useOpenClawSettings` -> `useChatBackendSettings`
2. `openClawSettings` -> `chatBackendSettings`

先保留旧导出别名，避免一次性改太多文件。

### 11.3 错误归一化

扩展 `normalizeErrorMessage.js`：

1. 保留 `openclaw_*`。
2. 增加 `nanobot_*` 翻译映射。

## 12. 测试计划

## 12.1 单元测试（必须新增）

桌面端：

1. `backendManager` 选择逻辑测试。
2. `settingsStore` 旧结构迁移测试。
3. `chatStream` 在两种 backend 下的事件行为测试。
4. `nanobotBackend` 错误映射测试。

前端：

1. 设置 payload 构建测试（按 backend 分支）。
2. 错误码翻译测试（新增 `nanobot_*`）。

## 12.2 集成测试（建议）

1. OpenClaw 模式回归测试：发送、流式、停止、错误提示。
2. Nanobot 模式联调：连接测试、发送、取消、多轮会话。

## 12.3 手工验收用例

1. 首次安装后默认仍能用 OpenClaw。
2. 切换 Nanobot 并配置 provider key 后可对话。
3. 切换回 OpenClaw 后历史功能不回归。
4. 断网/错误 key 时错误提示正确。

## 13. 分阶段落地计划

### Phase A：解耦重构（不改功能）

1. 引入 `ChatBackendAdapter` 抽象。
2. 将现有 OpenClaw 逻辑迁移到 `openclawBackend.js`。
3. 所有测试保持通过。

### Phase B：Nanobot MVP

1. 增加 settings 新结构与迁移。
2. 增加 `nanobotBackend + nanobot_bridge.py`。
3. 打通最小聊天链路（可先粗粒度流式）。

### Phase C：体验与稳定性

1. 增强取消和超时机制。
2. 优化 `text-delta` 输出颗粒度。
3. 补齐错误码、i18n、日志可观测。

### Phase D：发布准备

1. 打包 Python runtime 与 nanobot 依赖。
2. 新增安装后健康检查与诊断页。
3. 完成回归测试矩阵。

## 14. 打包与运行时建议

参考项目现有“内置 Python 语音栈”能力，复用同类策略：

1. 应用首次启用 Nanobot 时按平台下载 Python runtime。
2. 在应用私有目录安装 nanobot 依赖。
3. 运行 `nanobot_bridge.py` 时固定使用该私有 Python。
4. 缓存 runtime 与依赖，避免重复安装。

这样用户无需手工安装 Python 或 OpenClaw。

## 15. 关键风险与应对

1. 风险：nanobot 工具权限过大。
   应对：默认受限 workspace，禁用高风险工具，加入权限提示。

2. 风险：首次启动慢（安装依赖）。
   应对：异步安装、进度条、后台预热、失败可重试。

3. 风险：取消不及时。
   应对：请求级取消 + 进程级兜底终止。

4. 风险：兼容旧配置失败。
   应对：settings 迁移测试 + 回滚机制 + 版本字段。

## 16. 验收标准（DoD）

满足以下项视为接入完成：

1. 设置页支持切换 `OpenClaw / Nanobot`。
2. Nanobot 模式下可在无 OpenClaw 情况完成聊天。
3. 两种模式都能发出 `text-delta / done / error`。
4. 中断可用，异常有可读错误提示。
5. OpenClaw 旧功能无回归。
6. 自动迁移旧配置且不丢失 token。
7. 单元测试和核心集成测试通过。

## 17. 建议的首个实现里程碑

建议先交付一个“最小可用里程碑（M1）”：

1. 完成后端抽象和 settings 迁移。
2. Nanobot 完成“非 token 级流式”的稳定文本对话。
3. 打包内先支持开发模式 runtime（系统 Python）验证链路。
4. 验证通过后再做“内置 runtime 全自动安装”。

这样能最快验证方向正确，降低一次性重构风险。
