# Free-Agent VTuber OpenClaw 语音稳定性与防闪退加固方案（2026-03-04）

日期：2026-03-04  
适用范围：当前 Electron 主进程内运行 ASR/TTS provider（`sherpa-onnx-node`）的桌面版本

## 1. 背景

近期语音链路在“开始会话 / 提交输入”场景出现两类问题：

1. 前端状态错乱：VAD 已启动，但会话按钮状态异常（看起来像会话未启动）。
2. 提交输入后进程退出：`desktop:dev:electron` 直接退出，表现为应用闪退。

第二类问题属于稳定性高风险问题，需要专项治理。

## 2. 本次问题复盘（2026-03-04）

### 2.1 现象

日志出现：

- `offline-recognizer-impl.cc:Create:309 Please provide a model`
- Electron 进程以非 0 码退出（开发环境中 `desktop:dev:electron exited with code 255`）

### 2.2 根因

1. 内置 ASR 包使用 `zipformer2ctc`，但运行时默认推断为 `offline` 模式。  
2. `sherpa-onnx-node` 为原生模块，配置错误触发 native abort。  
3. ASR/TTS 在主进程内执行，native abort 会直接带崩 Electron 主进程，JS `try/catch` 无法兜底。

### 2.3 已完成修复

1. 修复前端会话面板生命周期错误，避免会话状态被误清理。  
2. ASR 模式推断修复：`modelKind=zipformer2ctc` 时默认走 `online`（除非显式覆盖）。  
3. 增加回归测试，验证 `zipformer2ctc` 的模式推断行为。

## 3. 现状评估

当前版本已修复已知触发点，但“主进程内调用原生 ASR/TTS”架构仍存在系统性风险：

1. 任意未知 native 异常都可能导致整应用退出。  
2. 主进程与语音计算耦合，无法将崩溃影响限定在语音子系统。  
3. 异常恢复能力不足（无法做到崩溃后自动拉起并恢复到可用状态）。

## 4. 目标与非目标

### 4.1 目标

1. 语音模块异常不再导致桌面应用整体退出。  
2. 配置错误在“提交输入前”可被识别并阻断。  
3. 出错时给用户明确反馈（错误原因 + 可执行操作）。

### 4.2 非目标（本阶段不做）

1. 不做完整语音链路重构。  
2. 不引入新的云端语音后端。  
3. 不改变当前 UI 交互主流程（仅做稳定性增强）。

## 5. 加固路线（建议分阶段）

### Phase A：快速防护（短期，1-2 天）

1. 在会话启动或提交前做 ASR/TTS 配置预检：
   - 模型文件存在性
   - `modelKind` 与 `recognizerMode` 兼容性
   - provider 必填项完整性
2. 预检失败时阻止提交，返回结构化错误，不进入 native 调用。
3. 在设置面板增加“语音环境自检”入口（或启动时自动自检并展示结果）。

验收：

1. 缺失/错配模型时，不会触发进程退出。  
2. 用户可在 UI 中看到明确失败原因。

### Phase B：进程隔离（中期，2-4 天）

1. 将 ASR/TTS provider 执行迁移到独立进程（`utilityProcess` 或 `child_process`）。
2. 主进程仅负责：
   - 会话状态机
   - IPC 转发
   - 子进程生命周期管理（拉起、心跳、重启）
3. 子进程异常退出时：
   - 主进程捕获退出事件
   - 对当前会话回送 `voice:event(type=error)`
   - 保持主窗口与主进程存活

验收：

1. 强制模拟语音子进程崩溃，主应用不退出。  
2. 崩溃后可自动恢复到可重新开始会话的状态。

### Phase C：可观测性与回归保障（后续）

1. 增加稳定性指标：
   - `voice_provider_crash_count`
   - `voice_session_error_rate`
   - `voice_session_recover_success_rate`
2. 增加稳定性回归测试：
   - 错配模型配置
   - provider 初始化失败
   - 子进程意外退出恢复

## 6. 关键设计要点（进程隔离）

1. 语音子进程协议最小化：只传递必要音频/文本/状态，不暴露敏感设置。  
2. 子进程无权限直接操作 UI。  
3. 主进程对所有子进程返回数据做 schema 校验，防止脏数据污染会话状态机。  
4. 子进程设超时与心跳，避免无响应卡死。

## 7. 风险与取舍

1. 进程隔离增加实现复杂度（协议、生命周期、测试成本）。  
2. IPC 转发会有少量时延开销。  
3. 但相比“整应用闪退”，该取舍是必要且高收益的。

## 8. 建议优先级

1. 先做 Phase A（预检 + 可见错误），立刻降低错误触发率。  
2. 再做 Phase B（隔离执行），解决“闪退级”风险根因。  
3. 最后补 Phase C（指标与自动化回归），防止回归。

## 9. 里程碑定义

1. M1（可发布）：错配配置不闪退，错误可见。  
2. M2（稳定版）：语音 provider 崩溃不影响主应用存活。  
3. M3（长期维护）：有指标、有告警、有自动回归用例。
