# Qwen3-TTS VoiceDesign 热态延迟与内存测试报告（2026-03-06）

## 1. 测试目标

验证 `Free-Agent-Vtuber-Openclaw/.tmp` 目录下已配置完成的阿里 Qwen 3 TTS，在非冷启动状态下：

1. 10 个汉字文本的 TTS 延迟大约是多少
2. 模型运行时会占用多少内存

本次测试仅针对 TTS 本体，不包含 Electron UI、主工程语音会话 IPC、音频播放链路等额外开销。

## 2. 测试环境

- 机器：`MacBook Air`
- 芯片：`Apple M4`
- CPU 核心：`10` 核（`4` 性能核 + `6` 能效核）
- 内存：`24 GB`
- Python：`3.11.14`
- 运行环境目录：`Free-Agent-Vtuber-Openclaw/.tmp/mlx-audio-test`
- 虚拟环境：`Free-Agent-Vtuber-Openclaw/.tmp/mlx-audio-test/.venv`
- TTS 框架：`mlx-audio`
- 模型：`Qwen3-TTS-12Hz-1.7B-VoiceDesign-bf16`
- 模型目录：`Free-Agent-Vtuber-Openclaw/.tmp/mlx-audio-test/models/Qwen3-TTS-12Hz-1.7B-VoiceDesign-bf16`

## 3. 测试输入

- 文本：`今天我们测试延迟表现`
- 文本长度：`10` 个汉字
- `instruct`：`自然、清晰、温和语气`
- `lang_code`：`Chinese`

## 4. 测试方法

为避免把 Python 进程重复启动时间混入“非冷启动”结果，本次测试使用同一个 Python 进程，按以下顺序执行：

1. 先调用 `load_model()` 加载模型
2. 执行 `1` 次预热推理
3. 在模型已驻留内存的情况下连续执行 `5` 次同文本推理
4. 记录：
   - 模型冷加载耗时
   - 预热耗时
   - 每次热态推理首段返回时间
   - 每次热态总耗时
   - 进程 RSS 常驻值与峰值
   - `mlx-audio` 内部上报的峰值内存

说明：

- 该模型在本次测试中每轮只返回 `1` 段音频，因此“首包延迟”基本等于“整句完成时间”。
- RSS 使用 `ps -o rss` 轮询采样，属于操作系统进程口径。
- `mlx-audio` 的 `peak_memory_usage` 属于框架内部统计口径，数值可能高于进程 RSS。

## 5. 模型体积

- 模型目录总大小：`4.2 GB`
- 主权重 `model.safetensors`：`3.6 GB`
- `speech_tokenizer/model.safetensors`：`651 MB`

## 6. 结果

### 6.1 冷加载与预热

- 冷加载耗时：`4115.78 ms`
- 加载完成后 RSS：`4047.20 MB`
- 预热首轮耗时：`4184.82 ms`
- 预热音频时长：`3.04 s`
- 预热完成后 RSS：`3306.95 MB`

| run_2 | `2818.01 ms` | `2825.02 ms` | `2.08 s` | `6.43 GB` |

## 7. 内存占用结论

按本次测试的两种统计口径：

- 进程空载基线：`58.48 MB`
- 模型加载后 RSS：约 `4.0 GB`
- 预热后稳定 RSS：约 `3.3 GB`
- 测试期间进程峰值 RSS：约 `4.32 GB`
- `mlx-audio` 内部上报峰值内存：`6.43 GB ~ 6.87 GB`

可操作性的理解方式：

- 常驻占用可按 `3.3 GB` 级别估算
- 运行峰值可按 `4.3 GB RSS` 级别估算
- 若按框架内部统计口径，推理阶段峰值接近 `6.9 GB`

## 8. 结论

本次测试下，`Qwen3-TTS-12Hz-1.7B-VoiceDesign-bf16` 在 `Apple M4 / 24 GB` 环境中，热态下生成 `10` 个汉字的 TTS：

- 典型耗时约为 `3` 秒
- 波动时会升到 `4` 到 `5.3` 秒
- 并不属于“低延迟可实时对话”的表现

如果按产品体验来判断，这个热态性能更接近“可本地运行、但响应偏慢”，而不是“本地实时 TTS”。

## 9. 备注

测试过程中出现了两个运行时警告，但本次未阻止模型正常出声：

1. `qwen3_tts` model type 配置警告
2. tokenizer `regex pattern` 警告
