# Edge-TTS 云端热态延迟测试报告（2026-03-06）

## 1. 测试目标

在 `Free-Agent-Vtuber-Openclaw/.tmp` 目录下部署 Python 生态的 `edge-tts`，并验证其在云端 TTS 模式下：

1. 10 个汉字文本的 TTS 延迟大约是多少
2. 采用中文女声、自然参数时的首段返回时间和整段接收时间表现如何

本次测试仅针对 `edge-tts` 云端语音服务本体，不包含 Electron UI、主工程 IPC、音频播放链路等额外开销。

## 2. 部署结果

- 测试目录：`Free-Agent-Vtuber-Openclaw/.tmp/edge-tts-test`
- 虚拟环境：`Free-Agent-Vtuber-Openclaw/.tmp/edge-tts-test/.venv`
- Python：`3.14.2`
- `edge-tts`：`7.2.7`
- 输出目录：`Free-Agent-Vtuber-Openclaw/.tmp/edge-tts-test/out`

## 3. 测试环境

- 机器：`MacBook Air`
- 芯片：`Apple M4`
- CPU 核心：`10` 核（`4` 性能核 + `6` 能效核）
- 内存：`24 GB`
- TTS 类型：`云端`

## 4. 语音参数

为满足“女声 + 自然声音”的要求，本次测试使用：

- Voice：`zh-CN-XiaoxiaoNeural`
- Gender：`Female`
- Voice 风格标签：`Warm`
- 语速：`+0%`
- 音调：`+0Hz`
- 音量：`+0%`

选择理由：

- `zh-CN-XiaoxiaoNeural` 是标准中文女声
- 不额外抬高语速、音调、音量，避免人为制造“播报感”或“夸张感”
- 作为默认自然参数的基线更适合做性能对比

## 5. 测试输入

- 文本：`今天我们测试延迟表现`
- 文本长度：`10` 个汉字

## 6. 测试方法

为尽量贴近“非冷启动”口径，本次测试使用同一个 Python 进程，按以下顺序执行：

1. 先发起 `1` 次预热请求
2. 然后连续执行 `5` 次正式请求
3. 每次请求通过 `edge_tts.Communicate(...).stream()` 流式接收音频
4. 记录两个延迟指标：
   - 首个音频 chunk 到达时间
   - 整段音频流接收完成时间

说明：

- `edge-tts` 属于云端服务，本次结果包含网络往返、服务端合成、流式回传三部分开销
- 本次每轮都收到了 `23` 个音频 chunk，外加 `1` 个元数据事件
- 每轮音频文件均成功输出为 `.mp3`

- 音频 chunk 数：`23`
- 音频大小：`16128 bytes`

## 9. 结论

本次测试下，`edge-tts` 在 `Apple M4 / 24 GB` 环境中，使用 `zh-CN-XiaoxiaoNeural` 中文女声、默认自然参数时，热态下生成 `10` 个汉字：

- 首个可播放音频通常在 `1.1` 到 `1.6` 秒到达
- 整段音频通常在 `1.4` 到 `2.1` 秒内接收完成
- 相比前一份本地 `Qwen3-TTS-12Hz-1.7B-VoiceDesign-bf16` 测试，云端 `edge-tts` 延迟明显更低

如果只看交互响应速度，当前这次 `edge-tts` 的体验明显更接近可用的在线 TTS。

## 10. 备注

由于该方案依赖云端服务，本次数据会受到以下因素影响：

1. 当前网络状态
2. 微软语音服务侧瞬时负载
3. 地域链路延迟

因此这份结果适合作为当前环境下的经验值，不应视为绝对恒定延迟。
